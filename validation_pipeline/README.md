## Validation Pipeline

Wearonomics processes wearable movement data through explicit, rule-based validation.

Raw datasets are ingested from wearable devices and parsed into time-sequenced observations. Each observation is evaluated using defined thresholds such as speed, continuity, and temporal consistency.

The pipeline distinguishes between valid locomotion and non-qualifying segments, including pauses, transport artifacts, and irregular patterns.

Movement is segmented into coherent intervals, allowing clear identification of active walking periods. Anomalies are not removed silently. They are detected, flagged, and excluded through transparent rules.

All outputs are reproducible. Given the same dataset and parameters, the system produces identical results.

This ensures that validation remains explainable, auditable, and suitable for research use.


## Validation Methodology

The Wearonomics Lab is structured around a validation-first pipeline, where every dataset is processed through a clearly defined sequence of transformations before any interpretation or economic value is assigned. At a high level, the system follows a deterministic flow: raw wearable datasets are ingested, converted into time-sequenced observations, segmented into micro-movements, evaluated through explicit behavioural rules, and finally classified into walking, stationary periods, transport, or artifacts. This entire process is implemented through a compact but well-defined code structure across the `validation_pipeline` and `src` directories.

The core of the validation logic resides in `src/lib/csv-parser.ts`. Despite its name, this file acts as the central validation engine. It defines the fundamental `Point` structure, which standardises all incoming data regardless of source format. Each point includes timestamp, latitude, longitude, elevation, speed, cadence, steps, distance, and heart rate. More importantly, it defines the `LedgerSegment` structure, which represents the atomic unit of validation. Each segment captures the transition between two consecutive points and stores start and end timestamps, duration, distance, speed, vertical variation, slope, heart rate, movement mode, classification, and confidence. This design ensures that validation is performed at the level of movement itself, rather than at the level of the full activity, creating a fully auditable trail of decisions.

The pipeline begins at ingestion through the application interface in `src/app/page.tsx`. The system accepts CSV, GPX, and TCX files and passes them into `addUpload()` in `src/lib/library.ts`. This function reads the file, extracts metadata, and stores it in the local dataset library. Crucially, ingestion immediately triggers `rebuildLedgerFromLibrary()`, which reprocesses all stored datasets through the validation pipeline. This design guarantees reproducibility, as validation and credit attribution are always derived from the same deterministic process rather than incremental updates.

Parsing is format-specific but converges into a unified structure. GPX and TCX files are parsed through XML extraction routines that recover timestamps, coordinates, elevation, and heart rate signals. These values are normalised, sorted chronologically, and used to derive speed either from recorded distance or through geographic calculations such as the haversine formula. CSV ingestion follows a simpler path, typically requiring timestamp, latitude, and longitude, with other variables optionally populated. Regardless of origin, the output of this stage is always a normalised array of `Point[]`, which becomes the input for validation.

Segmentation and validation are executed through `segmentAndValidate(points)`, which loads behavioural parameters from `src/lib/controls.ts` and passes the dataset into `runValidation()`. The `controls.ts` file defines the human validation profile, including key thresholds such as walking speed ranges, GPS artifact limits, transport detection triggers, and temporal conditions required for state transitions. For example, walking is defined within a speed band of approximately 0.5 to 8.0 m/s, GPS anomalies are flagged above 15.0 m/s, and transport classification is triggered only after sustained high-speed movement above 3.5 m/s for a defined duration.

The validation engine operates by iterating through each pair of consecutive points and constructing a segment for each transition. For every segment, it calculates duration, distance, speed, elevation change, slope, and average heart rate. Each segment initially exists in an undefined state and is then classified through rule evaluation. Segments exceeding the GPS threshold are marked as `gps_artifact`. Segments that sustain high speeds over time trigger a transition into `CAR` mode and are classified as `non_human_transport`. Low-speed persistence results in `stationary` classification, while segments within the locomotion band are marked as `walking`. Segments that do not meet any criteria remain unclassified.

A defining characteristic of this pipeline is its stateful logic. The system maintains internal variables such as `currentMode`, `highSpeedSeconds`, `lowSpeedSeconds`, `stoppedSeconds`, `walkBlockSeconds`, and `inLocomotionBandSeconds`. This introduces temporal continuity into the classification process. A brief spike in speed does not immediately result in transport classification unless it is sustained, and a temporary slowdown during vehicle movement does not cause premature reclassification as walking. This mechanism directly addresses common edge cases, such as stop-and-go traffic, where naive threshold systems typically fail.

Following segmentation, the system aggregates all classified segments into a `ValidationReport`. This report includes total duration, validated walking time, stationary time, transport exposure, and excluded artifacts, along with a breakdown of classification categories. The report is not an abstract summary but a direct aggregation of the underlying segment ledger, preserving full traceability of how each metric was derived.

The reporting layer, implemented in `src/app/report/page.tsx`, transforms the validation output into minute-level intervals. Each minute is evaluated based on overlapping validated walking segments, with metrics such as average speed and heart rate calculated for that interval. Only minutes containing a minimum threshold of validated walking time, typically 30 seconds, are considered eligible for economic attribution. This additional filtering layer ensures that fragmented or incidental movement does not generate value.

Economic attribution is handled separately in `src/lib/economics.ts`. This module defines parameters such as `creditRatePerMinute`, daily limits, and potential decay functions. In the current implementation, eligible minutes are multiplied by a fixed rate, producing Wearonomics Credits. This separation between validation and economics is intentional, allowing the behavioural model and incentive structure to evolve independently without compromising the integrity of the validation logic.

The final stage of the pipeline is the ledger system, implemented across `src/lib/library.ts` and `src/lib/ledger.ts`. The library module orchestrates the full rebuild of validated datasets, while the ledger module defines the structure of each record. A `LedgerEntry` includes validated time, validated distance, average speed, transport exposure, artifact exclusion, pause count, heart rate range, and total credits generated. This transforms each dataset into a structured research artifact, where behavioural validation and economic attribution are recorded together in a consistent and auditable format.

In its current form, the validation pipeline represents a rule-based, explainable system for interpreting wearable data. The architecture is deliberately compact and transparent, prioritising clarity and reproducibility. At the same time, the design allows for future extensions, as additional signals such as cadence consistency, acceleration patterns, and gap detection can be integrated into the validation logic. This positions the Wearonomics Lab not only as a working system, but as a foundation for ongoing research into the reliable interpretation and valuation of human behavioural data.
